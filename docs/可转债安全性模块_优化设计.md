# 可转债安全性模块 · 优化设计

> 状态：开发框架已落地，等待真实数据源 API 字段样例完成最后适配  
> 适用项目：`portfolio-server`  
> 业务口径：沿用 `scripts/generate_excel_v2.py` 的三指标打分法

## 1. 对原开发文档的审阅结论

原文档的目标、14 个输出字段和三指标规则可以作为业务需求基线，但不适合直接按其中的技术方案实施。

主要问题：

1. **技术栈与现有项目重复。** 当前项目已有 Node.js/Express 服务、PostgreSQL、登录会话、Nginx 部署和独立 worker。再引入 FastAPI、Vue、Element Plus 和第二套容器，会形成两套鉴权、日志、配置、部署和故障处理链路。
2. **“缓存”定义不足。** 内存或单文件缓存会在重启、多实例或写入中断时失效，不适合当前公网服务。应使用 PostgreSQL 保存不可变的成功快照。
3. **读取与刷新语义混淆。** `GET /api/bonds` 不应每次调用上游；读取只访问最后成功快照，只有定时任务或受控的手动刷新才访问上游。
4. **手动刷新缺少权限与并发控制。** 任意登录用户都能调用会消耗上游额度。当前设计限定管理员刷新，并用 PostgreSQL 咨询锁保证多进程只有一个刷新任务。
5. **失败策略缺失。** 上游超时、空数组、字段变化、同名公司冲突时不应覆盖线上数据。新设计只保存通过校验的成功快照，失败自动继续提供上一份。
6. **数据契约不完整。** 原文档没有定义分页、超时、鉴权头、百分比单位、空值、时间格式、重复公司名及上游更新时间。这些必须在真实 API 对接前确认。
7. **算法存在已知数据质量风险。** 旧脚本在指标二中把缺失的现金、金融资产、流动负债和有息负债当作 0，可能使该指标意外得分。为保证结果可比，当前框架保留此行为，同时记录 `incomplete_companies`；后续若要改为“缺失即未评级”，必须作为业务规则版本升级，不能静默修改。
8. **公司名称精确关联较脆弱。** 简繁体、ST 前缀、空格或曾用名会导致“未评级”。当前先记录未匹配清单，后续应增加人工维护的名称映射表，而不是模糊匹配后静默挂错公司。
9. **里程碑顺序不合理。** 登录、定时刷新不应放在 M6，因为它们已是公网 MVP 的必要条件；现有项目已经提供这两项基础设施。

## 2. 最终技术方案

```text
浏览器（现有统一登录）
        │
        ▼
Express /api/bond-safety/*
        │ 读取
        ▼
PostgreSQL 成功快照（最近 30 份）
        ▲
        │ 原子写入
每日 worker / 管理员手动刷新
        │
        ▼
API 适配器 → 纯算法模块 → 数据校验与诊断
        │
        ▼
用户提供的数据源 API
```

市场数据缓存补充：

- `market_instruments`：持久化 Tushare 证券代码与名称，默认 30 天刷新；
- `market_quote_cache`：按 `(symbol, source)` 保存 Tushare 日行情和腾讯最新行情；
- 腾讯行情默认 60 秒 TTL，只批量请求缺失或过期代码；请求失败继续使用最后成功值；
- 用户已有的 `daily_prices` 与 `index_history` 继续负责账户日终价格和指数历史，不与全局行情缓存混用；
- 安全性页面只读取 `bond_safety_snapshots`，普通页面访问不会触发任何上游 API。

采用现有项目技术栈：

- 后端：Node.js 22 + Express；
- 数据：PostgreSQL JSONB 快照；
- 前端：当前项目的原生 HTML/CSS/JavaScript 单页体系；
- 鉴权：复用现有 Session 登录，读取需登录，手动刷新需管理员；
- 调度：复用 `server/worker.js`，每天上海时间 06:30；
- 部署：复用现有 PM2/Nginx，不新增端口和服务。

## 3. 模块边界

| 模块 | 职责 |
|---|---|
| `server/services/bondSafety.js` | 纯函数：清洗、三指标评级、精确匹配、排序、诊断 |
| `server/services/bondSafetyFetcher.js` | 上游鉴权、超时、返回包装解析；真实字段差异在此适配 |
| `server/services/bondSafetyService.js` | 任务互斥、刷新编排、保存/读取最后成功快照 |
| `server/jobs/bondSafetyRefresh.js` | 每日 06:30 调度和首次无数据补齐 |

| `public/js/bond-safety.js` | 本地搜索、筛选、排序、表格渲染 |

## 4. 内部数据契约

核心算法只接受原文档定义的英文标准字段。适配器输出：

```json
{
  "companyRows": [{
    "company": "起帆电缆", "industry": "电气设备", "has_cb": 1,
    "interest_expense": 100, "ebit": 800, "cash": 500,
    "trading_fin_assets": 50, "interest_bearing_debt": 300,
    "total_liability": 1000, "current_liability": 600, "market_cap": 1500
  }],
  "bondRows": [{
    "bond_code": "113000", "bond_name": "起帆转债", "stock_name": "起帆电缆",
    "pe_ttm": 12.3, "pb": 1.2, "dividend_yield": 0.01,
    "bond_price": 112.5,
    "change_pct": 0.005, "double_low": 130.2, "convert_premium": 0.051,
    "convert_price": 20.0, "convert_value": 110.0
  }],
  "sourceUpdatedAt": "2026-07-17T06:00:00+08:00"
}
```

对接真实 API 前必须确认：

- 一个聚合接口还是两个独立接口；
- Token 所在请求头以及是否带 `Bearer`；
- 分页方式、单页上限、频率限制、数据更新时间；
- 金额单位是否一致；
- `dividend_yield`、`change_pct`、`convert_premium` 使用小数比例还是百分点；
- `null`、空字符串、`NaN`、`=` 前缀的含义；
- 公司名是否与债券的正股名使用同一标准。

## 5. 接口

### `GET /api/bond-safety/bonds`

需登录。读取最后成功快照，不调用上游。可选 `rating=安全|低风险|中风险|高风险|未评级`。

### `POST /api/bond-safety/refresh`

仅管理员。同步执行一次拉取、计算和校验；若已有任务运行返回 `409`，未配置数据源返回 `503`，上游失败返回 `502` 且保留旧快照。

## 6. 环境变量

支持以下二选一方式：

```dotenv
# 方式一：聚合接口，同时返回 company_financial 和 bond_quote
BOND_SAFETY_API_URL=https://example.com/api/bond-safety

# 方式二：两个接口
# BOND_SAFETY_COMPANY_API_URL=https://example.com/api/companies
# BOND_SAFETY_QUOTE_API_URL=https://example.com/api/bonds

BOND_SAFETY_API_TOKEN=
BOND_SAFETY_API_AUTH_HEADER=Authorization
BOND_SAFETY_API_AUTH_SCHEME=Bearer
BOND_SAFETY_API_TIMEOUT_MS=15000
```

Token 只放服务器 `.env`，不得写入代码、前端或文档实例。

## 7. 上线前验收

1. 用真实 API 样例补齐 `bondSafetyFetcher.js` 字段适配及分页；

3. 检查未匹配正股、缺失财务字段和重复公司清单；
4. 确认百分比与金额单位后再决定页面格式；
5. 在测试环境执行数据库迁移、管理员刷新、定时刷新和上游失败回退；
6. 验证普通用户不能手动刷新，但能读取最后成功快照。
