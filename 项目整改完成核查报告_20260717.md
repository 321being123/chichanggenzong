- `npm audit --audit-level=high` 退出码为 0。

### P1-1 备份脚本的清理匹配不会命中实际文件

实际输出名：

```text
${PGDATABASE}_${STAMP}.sql.gz
```

清理模式却是：

```bash
*."${PGDATABASE}"_*.sql.gz*
```

模式在数据库名前额外要求一个点，例如 `.portfolio_`，与实际 `portfolio_` 不匹配，因此旧备份不会被删除。

此外，`*.sql.gz*` 会同时匹配数据、加密文件和校验文件；即使修正前导点，也会按“文件数量”而不是“备份组数量”保留 14 项，无法稳定保留 14 份完整备份。

#### 整改建议

按时间戳或备份基名管理一整组文件，先选出需要删除的备份基名，再同时删除对应 `.sql.gz`、`.gpg` 和 `.sha256`。

### P1-2 加密备份的校验与恢复说明不一致

当前顺序为：

1. 对明文 `.sql.gz` 生成 SHA256。
2. 加密为 `.sql.gz.gpg`。
3. 删除明文 `.sql.gz`。
4. 恢复说明把解密结果写到 `/tmp/restore.sql.gz`。
5. 执行原 `.sha256` 文件校验。

原校验文件记录的是已删除的原路径，恢复说明中的 `/tmp/restore.sql.gz` 路径与之不一致，`sha256sum -c` 不能按文档直接成功。

#### 整改建议

- 优先对最终 `.gpg` 文件生成 SHA256，恢复前先校验密文完整性，再解密。
- 如还需校验解密后的明文，应生成不依赖绝对/旧相对路径的校验值，并在恢复脚本中显式比较。
- 增加可自动运行的备份脚本测试和真实恢复演练记录。
- 生产环境应强制要求 `BACKUP_PASSPHRASE`，不能只“推荐填写”。
- `.gitignore` 增加 `backups/`、`*.sql.gz`、`*.gpg` 等防误提交规则。

### P1-3 测试入口可能产生假通过

#### 空库迁移

`migration.test.js` 在无法连接 PostgreSQL 时，把 SKIP 写入 PASS：

```js
results.push(['PASS', 'SKIP-空库迁移']);
```

因此统一入口显示 `migration.test.js ALL PASS`，但直接运行的真实输出是：

```text
[SKIP] 无可用 PostgreSQL / 无 CREATEDB 权限
```

#### Python 测试

`run-all.js` 遇到 `ModuleNotFoundError` 或 `ImportError` 时一律标记 SKIP，CI 也没有严格模式。如果 requirements 遗漏依赖或 venv 安装异常，CI 仍可能返回成功。

#### 整改建议

1. 本地允许 SKIP，但 CI 设置 `CI=1` 时，数据库或 Python 环境缺失必须失败。
2. SKIP 应单独计数，不能写入 PASS。
3. 空库测试应验证确实创建了临时数据库并执行过 `001_init`。
4. CI 最终汇总应显示测试数、通过数、失败数和跳过数，且生产阻断测试不允许跳过。

### P1-4 Python 综合回归依赖真实外部服务，CI 稳定性不足

`test_ipo_regression.py` 明确依赖：

- PostgreSQL。
- `TUSHARE_TOKEN`。
- Tushare/外部行情网络。
- 固定证券和历史数据。

这类测试直接放入必过 CI，可能因密钥不可用、PR 不传 secrets、上游限流、网络抖动或数据更新而失败。

建议拆分为：

- 必过的确定性单元测试：使用固定 fixture 和 mock。
- PostgreSQL 集成测试：只依赖 CI 临时数据库。
- 外部行情合约测试：定时运行或手工触发，不阻断普通 PR。

### P1-5 既有安全整改仍未闭环

以下项目在上一份报告中仍属待办，本轮代码没有完成：

1. 密码仍使用同步 PBKDF2 10,000 次，未迁移 Argon2id/scrypt。
2. `trust proxy` 默认仍为 1，而不是默认关闭、生产显式开启。
3. PostgreSQL SSL 仍使用 `rejectUnauthorized:false`。
4. 多个 500 路由仍直接返回数据库或内部 `e.message`。
5. 邮箱验证码仍只按当前 Session 限制 60 秒，没有 IP+邮箱 Redis 联合限流。
6. Excel 子进程 stderr 没有输出大小上限。

这些问题不一定阻断当前单机受控使用，但在公网、多用户或云数据库场景下仍应继续整改。

### P2-1 Python 依赖仍有两个事实源

项目同时存在：

- 根目录 `requirements.txt`：已固定版本，但缺少 `akshare` 和 `paramiko`。
- `ipo-report/requirements.txt`：包含实际脚本依赖，但全部未固定版本。

CI 安装根目录文件，部署文档或模块目录又可能安装另一份文件，仍无法保证全部 Python 脚本拥有一致、可复现的环境。

建议只保留一个权威依赖清单，覆盖全部运行脚本；测试、部署和开发环境全部从同一锁文件安装。

### P2-2 Docker 历史文档仍有陈旧描述

活跃 Docker 文件和代码已经清理，但 `CHANGELOG.md`、旧 PostgreSQL 交接文档和旧整改报告仍写有“Docker 可选”“Docker 配置已完成”等历史内容。

更新日志可以保留历史事实，但交接文档应增加“已废弃、当前不支持 Docker”的醒目标记，避免新维护者按陈旧步骤部署。
