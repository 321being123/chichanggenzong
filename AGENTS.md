# 项目后续功能开发强制约束

## 固定部署登录流程

- 项目采用四位应用版本号，当前版本为 `0.2.0.1`。统一以 `package.json` 的 `appVersion` 为线上代码版本和程序版本；npm 必需的 `version` 使用兼容写法 `0.2.0-1`。以后每次部署都必须更新 `appVersion`，用户未指定新版本时默认递增最后一位，并在部署后核对 `/health` 返回的版本号。

- 腾讯云生产服务器：`ubuntu@82.156.125.47`。`root` 账号不允许直接登录，不要再次尝试。固定读取项目根目录 `.env` 中的 `DEPLOY_SSH_PASSWORD`，通过 Paramiko 使用账号密码认证；执行部署命令时用同一密码通过标准输入完成 `sudo -S` 提权。不使用本机 SSH 私钥，也不把密码拼进命令参数。严禁输出密码，或将密码写入代码、文档及提交记录。
- GitHub：仓库 `git@github.com:321being123/chichanggenzong.git`，使用本机 `~/.ssh/id_ed25519` 和 `~/.ssh/known_hosts`。若直接 `git push` 出现主机密钥校验失败，使用：
  `GIT_SSH_COMMAND='ssh -i ~/.ssh/id_ed25519 -o UserKnownHostsFile=~/.ssh/known_hosts -o StrictHostKeyChecking=accept-new -o BatchMode=yes' git push origin master`
- 部署时先推送 GitHub，再按上述方式登录腾讯云，进入 `/opt/portfolio`，依次执行 `git pull --ff-only origin master`、`npm ci --omit=dev`、`pm2 restart portfolio-server --update-env`、`pm2 save`。重启后等待服务完成启动，再检查提交版本、PM2、`http://127.0.0.1:3000/health` 和本次相关接口。不要再尝试 `root`、私钥登录或要求用户重复输入密码。

以后新增或修改金融数据、数据库及产品功能时，必须同时满足以下要求：

## 1. API调用与增量更新

- 外部接口数据首次完整拉取后必须落库，页面和重复计算优先读取数据库，不能因用户查看、切换图表或刷新页面重复调用外部接口。
- 后续默认按同步游标、公告日期、交易日期或报告期增量拉取，只保留必要的短重叠窗口处理更正和漏数。
- 接口空值、异常值或失败不得覆盖数据库中的上一份有效数据。
- 异常数据优先进行一次定点重拉；仍异常时记录数据质量问题，禁止无限重试。
- 新接口接入前必须说明首次全量范围、日常增量范围、缓存周期、去重键、补漏方式和失败保留策略。

## 2. 数据库冗余与扩展

- 数据库设计必须考虑未来增加字段、市场、币种、数据源和历史版本，避免只为当前页面设计一次性表结构。
- 高频查询、筛选、排序和计算字段使用正式列；完整上游响应及暂未标准化字段保存在 `JSONB`，作为合理冗余和追溯依据。
- 原始数据、标准化事实、计算指标和页面快照分层保存，不能只保存最终页面JSON。
- 使用内部证券ID、公司ID和数据源ID关联，避免把单一市场代码或单一接口字段作为长期架构核心。
- 设计迁移时必须包含唯一约束、索引、单位、币种、时间口径、版本字段和数据质量约束。

## 3. 整体数据与产品架构

- 新增功能前必须检查它与现有采集、数据库、指标计算、接口、页面、任务调度、权限和缓存的关系，禁止孤立增加重复链路。
- 优先复用现有公共组件、数据源适配器、同步游标、质量记录和指标定义，不重复实现相同能力。
- 新功能不得破坏持仓、交易、打新日历、可转债、用户隔离及其他现有业务。
- 数据结构和功能代码必须支持兼容扩展；涉及替换旧结构时，先完成新链路验证，再进行明确范围的迁移或删除。
- 每次交付必须验证：增量更新、数据去重、失败保留、历史追溯、接口兼容、数据库迁移及相关自动测试。

## 4. 单个案例与全局同类问题核查

- 用户以某一只转债、股票、基金、某一条数据、某个页面、某项功能或其他单个对象作为问题案例时，该案例仅用于定位问题，不代表修复范围只限该对象。
- 修复后必须使用同一套规则核查其他持仓、自选、页面、功能及相关同类数据是否存在相同问题；适用时应进一步核查全部同类对象。
- 发现同类问题必须一并修复，不得等待用户逐只指出。
- 每次交付必须明确说明核查范围、通过数量及仍存在的异常；未完成同类核查不得宣布任务完成。

以上四类要求是后续功能开发的必检项，不得省略。
