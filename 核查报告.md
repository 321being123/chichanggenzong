# 持仓管理系统 — 代码核查报告

> 核查日期：2026-07-06
> 范围：server.js、server/db.js、public/shared/core.js、public/js/utils.js、index.html、login.html、enter_trade.js、各 fix_*.py / import_positions.py
> 结论：**架构可用但技术债明显；发现 1 个会静默丢数据的高危 bug、2 个功能失效 bug、多处冗余与死代码。**

---

## 一、架构问题

1. **数据双写，冗余且易不一致**
   `saveAccountData` 每次保存都把同一份数据既写进结构化表（positions/trades/cash_flows/nav_history），又整体 JSON 再写一份进 `account_data`。读的时候以结构化表为准、`account_data` 仅作兜底。双份存储增加写入量，也埋下"两份不一致"的隐患。

2. **core.js 宣称的"localStorage 双模式"未落地**
   core.js 头部注释说"被 仓位管理.html(localStorage) 和 index.html(fetch API) 共用"，但仓库里根本不存在 `仓位管理.html`。`loadData/saveData` 由 index.html 内联脚本注入为全局函数。这是误导性注释 + 不存在的架构分支。

3. **一次性 Python 修复脚本已失效且危险**
   `calc_cash.py / fix_currency.py / fix_hk_prices.py / fix_qty.py / fix_rate.py` 全部直接读写迁移前的旧 JSON 文件 `data/daicunzai__华泰账户.json`。系统早已迁移到 SQLite，这些脚本现在写的是一份"死"JSON，运行反而会让人误以为改了数据。属于应清理的堆积物。

4. **魔法数字散落**
   `2462607.72`（期初总市值）同时硬编码在 `index.html`(第277行) 和 5 个 Python 脚本里；HK 汇率在脚本间不统一（0.87 / 0.868 混用）。

5. **前端全局变量强耦合**
   `data / currentAccount / priceChangeMap / PRICE_CACHE / TOTAL_ASSET / accounts` 全是跨文件全局变量，core.js 依赖 index.html 注入的 `loadData/saveData`。没有模块边界，后期维护容易踩坑。

## 二、逻辑 Bug（按严重程度）

### P0 — 静默丢数据
**`confirmDelete()` 删除持仓会连带删除同代码的所有交易记录**
`core.js` 第 874 行：
```js
data.trades = data.trades.filter(t => data.positions.some(p => p.code === t.code));
```
删除某持仓后，系统把"代码不再存在于持仓里"的交易全部滤掉。例如手动删除"中国中车"持仓，会把它所有的买入/卖出交易历史也一起删掉。删除持仓不应抹掉交易流水（交易用于净值计算）。**必须修。**

### P1 — 功能失效
**1. 收益图"指数对比"开关全部失效**
`renderReturnsStats` 里按钮调用 `toggleIndex('sh'/'sz'/'zz'/'hs')`，但 `indexVisibility` 的键是中文（沪深300/上证指数/中证全指/恒生指数），数据集 label 也是中文。结果：`toggleIndex('sh')` 找不到对应数据集 → 点击毫无反应，4 条指数线永远全显示，无法隐藏。

**2. 收益图"时间区间"按钮高亮失效**
`switchPeriod(days)` 用 `.period-btn[data-days="..."]` 选元素加高亮，但那些按钮根本没有 `data-days` 属性 → active 样式永远不切换（区间切换功能本身生效，仅高亮错位）。

### P2 — 计算方法 / 体验
- **`recordNav` 现金流修正只用"当天"现金流**，不是"自上次净值以来的累计现金流"。若某些天没记录净值（没开 App / 收盘后），那天的出入金未参与修正，净值轻微失真。
- **亏损金额不显负号**：`renderReturnsStats` 用 `fmt(Math.abs(totalReturn))` 展示，亏损只靠红色提示，没有"-"号，数字本身是正的。
- **代码分类逻辑四处重复且不完全一致**：`utils.recognizeCode`、`utils.getSecId`、`server.fetchQuoteByCode`、`enter_trade.inferSubtype` 各写一套（前缀规则 / f57 判断混用），长期易漂移。
- **`refreshAllPrices` 与 `doRefresh` 双重 `saveData`+双重 `renderAll`**，每次刷新多一次无谓写入与重绘。
- **`init()` 中 `initAutoRefresh()` 被调用两次**（init 第 293-294 行），重复设置定时器（虽被 clear 抵消，属冗余）。

## 四、安全 / 健壮性（顺带，风险均低）

- 登录失败锁定 `failMap` 为内存态，进程重启即失效（个人本地应用可接受）。
- CSRF 用 Origin/Referer 校验，localhost 信任模型下可被同机其他 localhost 服务绕过，本地使用风险低。
- 会话 secret 已持久化到 `data/.secret`，OK。
- 用户数据按 `req.session.user` 隔离（load/save 都带 username），跨用户访问已被挡住，这点做得对。

---

## 五、优先修复建议

1. **【必修】** `confirmDelete` 删掉"连带删交易"那一行，仅删持仓、保留交易流水。
2. **【必修】** 统一收益图控件：`indexVisibility` 键与按钮参数对齐（都用中文键或都加 `data-idx`），并给区间按钮加 `data-days` 属性。
3. **【建议】** 删除失效的 Python 旧 JSON 脚本，统一 HK 汇率常量；仅保留可走 SQLite 的 `import_positions.py`。
4. **【建议】** 删除 `index.html` 的 `exportExcel()` 死函数；`init()` 里去掉重复的 `initAutoRefresh()`。
5. **【建议】** 把"代码→品种"分类收敛为单一函数，前后端共用，消除 4 处规则漂移。
