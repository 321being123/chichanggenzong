> 检查日期：2026-07-16  
> 检查范围：`portfolio-server` 全项目  
> 报告性质：代码、数据正确性、安全、依赖、性能、测试、部署与运维综合检查  
> 说明：本报告不包含任何真实密码、令牌或密钥值。

## 一、执行摘要

项目已经具备基本的模块化后端、PostgreSQL 数据存储、安全中间件、自动任务及一定数量的回归测试。现有可独立运行的安全专项、净值数学和 IPO API 测试均通过，说明部分核心功能已有较好的回归保护。

但本轮检查同时发现多项会影响生产安全或财务数据正确性的高风险问题，包括：数据库密码进入 Git、Docker 镜像可能打包真实 `.env`、混合持仓漏记可转债/ETF 收盘价、容器时区导致任务错时、Markdown 存储型 XSS，以及存在已知高危漏洞的 Excel 解析依赖。

综合判断：项目当前适合继续开发和受控验证，但在 P0 问题完成整改并通过回归验收前，不建议按“生产安全、财务数据可信”的标准直接上线或扩大用户范围。

## 三、P0：必须立即整改

### P0-1 数据库密码已进入 Git

#### 现状

8 个已纳入版本管理的 Python 脚本包含相同的非空硬编码数据库密码。当前本地分支与 `origin/master` 同步，应按凭据已经泄露处理，而不能仅删除当前代码中的字符串。

涉及文件包括：

- `ipo-report/backfill_bond_firstday.py`
- `ipo-report/backfill_bond_history.py`
- `ipo-report/backfill_bond_ratings.py`
- `ipo-report/backfill_lottery_rate.py`
- `ipo-report/fill_ratings_batch.py`
- `ipo-report/generate_server_bond_sync.py`
- `ipo-report/generate_server_ipo_reports_sync.py`
- `ipo-report/generate_server_ipo_sync.py`

#### 风险

- 未授权连接或破坏数据库。
- 持仓、交易、用户资料等敏感数据泄露。
- 即使当前代码删除密码，Git 历史仍可恢复旧值。

#### 整改措施

1. 立即轮换对应数据库用户密码。
2. 检查该账号的连接日志、失败登录和异常 SQL。
3. 所有脚本统一通过 `DATABASE_URL` 或 `PG*` 环境变量读取凭据。
4. 使用 `git filter-repo` 或 BFG 清理全部历史提交中的密码。
5. 强推前通知所有协作者，强推后要求重新克隆或清理本地历史。
6. 在 GitHub 开启 Secret Scanning，并在 CI 增加 Gitleaks/TruffleHog。

#### 验收标准

- 原密码无法连接数据库。
- 当前代码和完整 Git 历史均扫描不到该密码。
- CI 能自动拦截新增硬编码凭据。

### P0-2 Docker 镜像可能包含真实密钥

#### 现状

项目没有 `.dockerignore`，Dockerfile 使用 `COPY . .`。项目根目录存在已填写的 `.env`，其中包括数据库、AI、SMTP、Tushare 和管理员凭据，因此本地构建可能将其写入镜像层。

#### 风险

- 镜像上传仓库或交付他人后泄露全部环境密钥。
- 即使后续删除文件，旧镜像层仍可能包含秘密。
- `.git`、日志、数据库、测试数据和调试文件也会进入镜像。

#### 整改措施

1. 新增 `.dockerignore`，至少排除：

   ```text
   .git
   .github
   .env
   .env.*
   node_modules
   *.log
   data
   __pycache__
   *.pyc
   *.db
   ipo-report/history_reports
   ipo-report/individual
   debug_*
   test*.js
   nul
   ```

2. Dockerfile 先复制 `package.json` 与 `package-lock.json`，使用 `npm ci --omit=dev`。
3. 使用非 root 用户运行 Node。
4. 所有秘密在容器运行时通过环境变量或 Secret Manager 注入。
5. 若旧镜像曾构建、上传或共享，删除旧镜像并轮换其中全部凭据。

#### 验收标准

- `docker history` 和导出的镜像文件中不存在 `.env` 或秘密。
- 镜像只包含运行时必需文件。
- 容器以非 root 用户运行。

### P0-3 混合持仓漏记可转债/ETF 收盘价

#### 现状

`server/jobs/marketClose.js` 只要发现账户当天已有任意一条 `daily_prices` 记录，就跳过整个市场任务。A 股先写入数据后，可转债和 ETF 任务会被错误跳过；补漏逻辑也因已有任意记录而不再修复。

#### 影响

- 可转债、ETF 等资产缺失每日收盘价。
- 净值、总资产、区间收益和回撤可能不准确。
- 历史数据可能已经存在系统性缺口。

#### 整改措施

1. 按账户、日期、市场和证券代码查询缺失项，不能以“当天任意一条记录”作为完成条件。
2. 对持仓代码去重后，仅抓取缺失代码。
3. 补漏任务比较“应有代码集合”和“已有代码集合”。
4. 增加以下测试：
   - 仅 A 股账户。
   - 仅可转债账户。
   - A 股+可转债+ETF 混合账户。
   - 某一市场部分代码已写入、部分代码缺失。
5. 编写历史审计脚本，找出持仓存在但收盘价缺失的日期和代码，并进行可控回填。

#### 验收标准

- 混合账户每个交易日的全部在仓代码均有价格记录，或有明确失败状态。
- 补漏任务可修复部分缺失，不会因已有一条记录而跳过。
- 回填前后净值差异有审计报告。

### P0-4 容器时区可能导致任务错时

#### 现状

收盘任务使用 `Date.getDay()`、`setHours()` 等服务器本地时间 API，但 Docker/Compose 未显式配置 `Asia/Shanghai`。Alpine 容器通常采用 UTC，15:10 的任务可能在北京时间 23:10 执行。

#### 整改措施

1. 调度逻辑显式使用 `Asia/Shanghai`，不要依赖宿主机默认时区。
2. 容器增加 `TZ=Asia/Shanghai`，但业务代码仍应做时区显式化。
3. 增加 UTC、北京时间、跨日、周末、节假日和夏令时无关性测试。
4. 任务记录中同时保存计划时间、实际开始时间和业务日期。

#### 验收标准

- 无论宿主机为 UTC 还是东八区，任务都在预期北京时间执行。
- 日志能明确显示计划时间与实际偏差。

### P0-5 Markdown 报告存在存储型 XSS

#### 现状

`public/ipo-report.html` 将 `marked.parse()` 的结果直接赋值给 `innerHTML`。探针验证原始 HTML 事件属性会被保留；当前 CSP 还允许 `unsafe-inline`。

#### 整改措施

1. 使用 DOMPurify 等成熟库对 Markdown 生成的 HTML 做白名单净化。
2. 或禁用 Markdown 原始 HTML，只允许安全 Markdown 元素。
3. 错误消息通过 `textContent` 输出，不拼接进 `innerHTML`。
4. 逐步移除内联事件和内联脚本，收紧 CSP，去除 `unsafe-inline`。
5. 增加 `<script>`、`onerror`、`javascript:`、SVG 等 XSS 回归用例。

#### 验收标准

- 所有已知 XSS 测试载荷只能显示为文本或被移除。
- CSP 不再依赖 `script-src 'unsafe-inline'`。

### P0-6 Excel 解析依赖存在已知高危漏洞

#### 现状

`xlsx@0.18.5` 命中原型污染与 ReDoS 高危公告，npm 没有可用的自动修复版本。Excel 文件又属于用户可控输入，风险可被直接触发。

#### 整改措施

1. 评估迁移到维护中的 Excel 解析库或已修复的 SheetJS 发行渠道。
2. 解析放入独立 Worker/子进程，并设置 CPU、内存、文件大小和执行超时。
3. 校验文件魔数、压缩包展开大小、工作表数量、行列数和单元格长度。
4. 不要只校验压缩后文件大小，防止 ZIP Bomb。
5. 完成迁移前限制 Excel 导入入口的可用用户和调用频率。

#### 验收标准

- `npm audit` 不再报告该高危直接依赖。
- 恶意/畸形 Excel 无法拖垮 Web 主进程。

## 四、P1：近期整改

### P1-1 认证与会话安全

- 当前密码使用同步 PBKDF2 10,000 次，建议迁移到异步 Argon2id 或 scrypt。
- 使用 `crypto.timingSafeEqual` 比较哈希。
- 登录成功后调用 `req.session.regenerate()`，避免会话固定。
- 登出应等待 `session.destroy()` 回调完成后响应。
- 用户名和密码增加最大长度、字符规范与 Unicode 归一化。
- 登录失败统一返回模糊错误，避免枚举账号。
- 验证码改用 `crypto.randomInt()`。
- 登录、注册、验证码发送增加 Redis 分布式的账号/IP/邮箱联合限流。
- `require_email=1` 但邮件服务不可用时必须拒绝注册，不能静默绕过。

### P1-2 用户数据并发覆盖

`saveUsers()` 会遍历并更新整个用户集合。并发注册或修改账户列表时，旧快照可能覆盖其他用户的密码或账户信息。

建议：

- 为注册、账户列表修改和历史账户补登分别提供单用户 SQL。
- 注册使用 `INSERT ... ON CONFLICT` 或唯一键异常完成原子判重。
- 禁止普通业务流程读取全量用户名和密码哈希。
- `users.accounts` 与结构化 `accounts` 表确定唯一事实源。

### P1-3 服务端数据校验不完整

目前 `cashBase`、`hkRate`、顶层 `totalAsset` 和 `fundRecord` 没有完整 schema 校验，实测非法类型、负汇率和包含 HTML 的内容可以通过。

建议：

- 使用 JSON Schema、Zod、Joi 或 Ajv 建立完整请求模型。
- 对金额、汇率、数组大小、对象深度和文本长度设置明确边界。
- 对 AI 返回结果再次执行相同的服务端 schema 校验。
- 拒绝未知字段，避免 10 MiB 任意嵌套 JSON 落库。
- `version` 经过整数和范围校验后设为必填，移除绕过乐观锁的兼容路径。

### P1-4 行情缓存并发与上游风暴

行情缓存只保存最终 Map，没有保存正在进行的 Promise。冷缓存并发时可能重复请求 Tushare；账户读取还对全部代码直接执行无界 `Promise.all`。

建议：

- 实现 single-flight：同一缓存刷新期间所有请求复用一个 Promise。
- 对证券代码去重。
- 使用并发池限制外部请求数量。
- 对失败结果设置短时负缓存，防止重复打穿上游。
- 区分“缓存为空”和“缓存尚未加载”。

### P1-5 Redis 降级逻辑不真实

RedisStore 在连接结果确认前已经挂到 Session。连接失败时虽然打印“回退内存”，Session 仍可能持有不可用的 RedisStore。

建议：启动阶段等待 Redis 连接成功后再创建 Session Store；生产环境可直接失败启动，避免静默降级造成限流和会话保护失效。

### P1-6 代理、TLS 与错误信息

- `trust proxy=1` 不应在所有部署模式固定启用。
- Docker Compose 的 3000 端口应只绑定回环或内部网络。
- PostgreSQL SSL 不应使用 `rejectUnauthorized:false`，应配置 CA 并验证证书。
- 生产环境 500 响应不得返回原始数据库或内部错误信息。
- 内存限流 Map、登录失败 Map 和注册 IP Map 需要 TTL 清理，避免长期增长。

### P1-7 上传与内存控制

- 10 MiB 图片经 Base64 后约为 13.33 MiB，与 Express 10 MiB JSON 上限冲突。
- 图片 MIME 声明必须结合文件魔数校验。
- 二维码上传 token 创建与上传接口需要限流。
- 不应将多个大图片长期保存在 Web 进程内存 Map 中。
- 客户端不能任意指定高成本 AI 模型，应使用服务端白名单。

## 五、P2：工程质量与性能优化

### P2-1 测试体系

1. 将所有独立测试纳入统一入口：
   - 安全专项。
   - 净值数学。
   - IPO API。
   - IPO 前端。
   - Python 单元/回归。
   - PostgreSQL 集成测试。
2. 当前 IPO 前端有 2 项失败，修复或明确更新产品规则后才能放行。
3. `test:unit` 实际是依赖已启动服务和固定账号的端到端脚本，应重命名并隔离数据。
4. 为收盘价、调度时区、并发注册、乐观锁、Redis 故障和恶意上传增加回归测试。
5. 建立最低覆盖率门槛，优先覆盖资金、净值、交易和权限路径。

### P2-2 CI/CD

- CI 增加 ESLint、Prettier、Ruff、Python 类型检查和 Python 语法检查。
- 增加 `npm audit`、`pip-audit`、Secret Scanning 和容器扫描。
- 构建 Docker 镜像并检查镜像中不存在 `.env`、`.git` 和测试数据。
- GitHub Actions 使用最小权限，并考虑固定 Action 的完整提交 SHA。
- 设置分支保护：必须通过测试、安全扫描和代码审查后才能合并。

### P2-3 数据库与持久化

- 不要在每次启动时执行大量 `ALTER TABLE` 和类型转换。
- 引入有版本号、可审计、可回滚的迁移工具。
- 日期字段使用 `DATE/TIMESTAMPTZ`，金额保留 `numeric`。
- 为结构化子表增加外键与必要 CHECK 约束。
- `account_data` JSON 和结构化表长期双写会漂移，应逐步收敛为单一事实源。
- 缺少 `version` 的保存请求不应再允许无条件覆盖。

### P2-4 SQL 与接口性能

- `saveAccountData` 当前逐条写入，最大载荷可能产生上万次 SQL 往返；改为批量 INSERT、UNNEST 或 COPY。
- 多个互不依赖的账户读取查询可使用 `Promise.all` 并行。
- `saveDailyPrices`、`upsertIndexPoints` 改为批量 upsert。
- 为外部行情、指数和汇率增加合理缓存与熔断。
- 静态资源配置长期缓存和内容哈希；Nginx 开启 gzip/brotli。

### P2-5 代码结构

- `ipo_daily_report.py` 超过 4,300 行，建议拆分为：数据源适配、公告解析、分类、估值、预测、报告渲染和持久化。
- `server/db.js` 同时承担建表、迁移、用户、账户、行情、后台等职责，应按领域拆分 Repository。
- 前端逐步减少全局变量、内联脚本、内联样式和 `onclick`。
- 删除 `eval` 调用，使用标准模块导入。

### P2-6 依赖与可复现环境

- Python 依赖集中到 `pyproject.toml`，锁定版本并生成可复现锁文件。
- Node 保留 lockfile，Docker 使用 `npm ci` 而非 `npm install`。
- Express、Redis、connect-redis 等大版本升级需要单独分支和回归测试，不能一次性盲升。
- 补充 `engines`、许可证、仓库地址和运行时版本说明。

### P2-7 仓库卫生与文档

- 清理 Windows 特殊文件 `nul`，该文件会导致 `rg` 等工具报错。
- 明确 `ipo-report/data`、`history_reports`、`individual`、模型文件和 SQLite 数据库是源代码、构建制品还是运行数据。
- 生成物应写入对象存储或发布制品，不应随意混在工作树。
- 更新 README：当前数据源已是 PostgreSQL，不再是 `data/__users__.json`。
- 清理父目录中无效的 `.git` 和重复的根 `package.json/node_modules`，避免从错误目录执行命令。
- dangling Git blob 属低风险，可在确认无需恢复后通过常规 `git gc` 清理。

### P2-8 备份、监控与合规

- 将文档中的 `pg_dump` 落实为自动、加密、异地备份任务。
- 定期执行恢复演练，而不是只确认备份文件存在。
- 明确 RPO、RTO、保留周期和删除流程。
- 对任务失败、数据库不可用、收盘价缺失、登录暴力破解和 AI 成本异常建立告警。
- 投资持仓、交易、邮箱和头像属于敏感数据，应明确访问控制、保存期限和日志脱敏规则。

## 七、上线阻断条件

以下任一项未完成时，不建议扩大生产使用：

- 已泄露数据库密码尚未轮换。

- 混合持仓收盘价缺失问题未修复或历史数据未审计。

- Markdown 存储型 XSS 未修复。
- 高危 Excel 解析依赖仍直接处理不可信文件且没有隔离措施。
- 关键测试存在失败或未纳入 CI。
