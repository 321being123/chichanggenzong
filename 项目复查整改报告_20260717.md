## 一、复查结论

## 三、P0：上线阻断问题

### P0-1 全新数据库执行首次迁移会失败

#### 现状

`server/db/migrations.js` 在首次迁移中调用：

- 第 164 行：`seedBrokers()`
- 第 212 行：`migrateAccountsTable()`

但该模块没有从 `server/db/brokers.js` 和 `server/db/accounts.js` 导入这两个函数。

模拟全新数据库执行 `migration001Init()` 的结果为：

```text
migration_probe=fail
queries_before_failure=23
ReferenceError: seedBrokers is not defined
```

现有数据库如果已经记录 `001_init`，可能不会重新进入该迁移，因此会掩盖问题；新部署、新数据库或清库重建会直接启动失败。

#### 整改建议

1. 在迁移模块显式导入 `seedBrokers` 和 `migrateAccountsTable`。
2. 检查导入关系，避免形成数据库模块循环依赖。
3. 增加真正从空 PostgreSQL 数据库执行 `initSchema()` 的测试。
4. 集成测试开始前清空数据库或使用随机临时数据库，确保不会复用已经应用的迁移记录。

#### 验收标准

- 空数据库能够完整执行 `001_init`。
- `brokers`、`accounts`、`job_runs` 等表及种子数据正确生成。
- 连续执行两次迁移保持幂等，不重复写入、不报错。

### P0-2 本机仍保留旧密码及完整旧仓库历史

#### 已完成部分

当前可达 Git 历史中未再检出旧密码，说明历史重写对正常分支和提交链已经生效。

#### 未完成部分

对象级扫描发现当前 `.git/objects` 中仍存在：

```text
不可达 commit：31
不可达 tree：46
不可达 blob：21
命中旧密码的不可达 blob：11
```

项目父目录还存在：

- `portfolio-backup-20260716.tar.gz`：包含完整 `.env`、旧 `.git`、reflog 和对象库。
- `scrub_secret.py`：脚本自身包含用于替换的旧密码。

上述文件继承了较宽的本机访问权限。即使 GitHub 上的正常提交链已经清理，本机用户、备份泄露或文件误传仍可能暴露凭据。

#### 整改建议

1. 最终仍应轮换数据库密码；在轮换前，旧密码应继续按已泄露凭据管理。
2. 确认远端所有分支和标签均已清理后，优先采用重新克隆干净仓库的方式替换旧本地克隆。
3. 若保留现有克隆，应在确认无待恢复工作后清理 reflog 和不可达对象；此操作会永久删除 Git 恢复点，执行前必须单独确认。
4. 删除或净化 `scrub_secret.py`，不得继续以明文保留旧密码。
5. 备份如无需保留应安全删除；如必须保留，应加密、收紧 ACL，并移出普通工作目录。
6. 在 CI 增加 Gitleaks、TruffleHog 或等效秘密扫描。

#### 验收标准

- 当前 Git 可达及不可达对象均不命中旧密码。
- 本机普通目录不存在含旧密码的清理脚本或未加密仓库备份。
- 原密码无法连接数据库。
- CI 能阻断新提交的硬编码凭据。

### P0-3 Excel ZIP Bomb 防护字段读取错误且内存限制未生效

#### 现状

ZIP 本地文件头的标准字段为：

- 偏移 18：压缩后大小。
- 偏移 22：解压后大小。

`server/services/excelParser.worker.js` 当前反向读取了这两个字段，并按错误字段累计解压体积、移动扫描偏移。

`server/test/excel.test.js` 的压缩炸弹测试也使用了相同的错误偏移，因此测试虽然通过，却没有验证真实 ZIP Bomb 格式。

使用正确字段构造的 300MB 解压声明探针结果为：

```text
correct_zip_bomb_probe=rejected
rejected_by_bomb_guard=false
error_class=generic_parser
```

这说明文件只是被普通解析器碰巧拒绝，并非由压缩炸弹保护拦截。合法但高压缩比的恶意工作簿仍可能进入存在高危漏洞的 `xlsx` 解析流程。

此外，`server/services/excelSafe.js` 启动子进程时没有传入 `--max-old-space-size`，注释所称的独立内存上限实际没有生效；标准输出字符串也没有大小上限。

#### 整改建议

1. 修正 ZIP 字段偏移和扫描步长。
2. 优先使用成熟 ZIP 库读取中央目录，正确处理数据描述符、ZIP64、重叠条目及异常目录。
3. 对无法静态确定解压大小的数据描述符文件采取保守拒绝或隔离解压策略。
4. 子进程通过 Node 启动参数设置真实内存上限。
5. 限制子进程 stdout、stderr 和输入 JSON 大小。
6. 保留超时强杀，同时增加异常退出码处理。
7. 迁移到维护中的 Excel 解析库，最终消除 `xlsx` 高危审计告警。

#### 验收标准

- 正确构造的 ZIP Bomb 在进入 `XLSX.read()` 前被拒绝。
- `npm audit --audit-level=high` 不再报告 `xlsx` 高危漏洞。
- 子进程内存、运行时间和输出大小均有可验证的硬限制。

## 四、P1：新增功能和数据正确性回归

### P1-1 IPO Python 模块拆分后市场温度状态不同步

#### 现状

拆分后的模块普遍使用 `from ipo_lib_xxx import *`。`ipo_lib_sector.py` 的温度检测函数会重新绑定：

- `_MARKET_TEMP`
- `_BOND_MARKET_TEMP`

`ipo_lib_valuation.py` 和 `ipo_lib_report.py` 在导入时保存的是旧字典引用，重新绑定后不会自动指向新字典。

运行时探针结果：

```text
shared_before_rebind=true
shared_after_rebind=false
valuation_stale_after_rebind=true
```

因此主流程即使检测出常温或冷市，估值和报告模块仍可能继续按默认热市计算，影响预测涨幅、评分、PE 修正和申购建议。

#### 整改建议

1. 避免星号导入，改为 `import ipo_lib_sector as sector` 并动态读取 `sector._MARKET_TEMP`。
2. 或使用专门的共享状态对象，通过 `clear()`/`update()` 原地修改，避免重新绑定。
3. 更推荐将市场温度作为显式参数传入估值和报告函数，减少隐藏全局状态。
4. 增加“检测冷市后估值模块和报告模块均读取冷市”的端到端测试。

#### 验收标准

- 温度检测结果在估值、预测、建议、Markdown 和 HTML 报告中保持一致。
- 模块加载顺序不影响计算结果。

### P1-2 数据目录在数据库模块拆分后偏移一层

#### 现状

`server/db/connection.js` 当前默认值为：

```js
path.join(__dirname, '..', 'data')
```

由于 `__dirname` 已从 `server` 变为 `server/db`，当前路径实际解析为 `server/data`，而不是原项目根目录下的 `data`。

已经生成：

```text
?? server/data/.secret
```

影响范围包括：

- 会话密钥读取与生成位置改变。
- 遗留 JSON 数据迁移读取错误目录。
- 原 `data/.secret` 可能被忽略，而新 `server/data/.secret` 未被 `.gitignore` 覆盖。

#### 整改建议

1. 默认路径修正为项目根目录 `data`。
2. 增加 `DATA_DIR` 解析测试，分别覆盖默认路径和环境变量覆盖。
3. 确认新旧 `.secret` 后只保留生产实际使用的密钥文件。
4. `.gitignore` 增加防御性规则，但不能用忽略规则替代路径修复。

#### 验收标准

- 默认 `DATA_DIR` 与整改前保持一致。
- 启动后不再生成 `server/data`。
- JSON 迁移和会话密钥均读取预期目录。

### P1-3 收盘任务首次调度正确，后续续排仍依赖宿主时区

#### 现状

首次调度使用显式东八区的 `msUntil()`，现有测试也只验证了该辅助函数。

任务执行后的下一次调度仍使用：

```js
nextDay.setDate(...)
nextDay.setHours(mkt.h, mkt.m, 0, 0)
```

这两个方法按照服务器本地时区工作。在 UTC 服务器上，首次可在北京时间 15:10 正确触发，但后续可能被续排到 UTC 15:10，即北京时间 23:10。

`pickMissingCodes()` 也没有对结果去重，同一代码在多个持仓项出现时可能重复请求行情。

#### 整改建议

1. 每次续排都重新调用显式东八区的时间计算函数。
2. 不再使用本地时区的 `setDate()`、`setHours()`。
3. 持仓代码使用 `Set` 去重后再抓取。
4. 在 `TZ=UTC` 下模拟首次触发、第二次触发、周末和节假日续排。

#### 验收标准

- 服务器处于 UTC、Asia/Shanghai 或其他时区时，连续多天均按北京时间触发。
- 同一账户、市场、日期和代码最多请求、写入一次。

### P1-4 IPO 报告错误和空状态处理会抛出二次异常

#### 现状

`public/ipo-report.html` 的 `setState()` 使用变量 `content`，但 `content` 仅通过 `var` 定义在 `init()` 函数内部。

401、空报告、HTTP 错误或解析异常进入 `setState()` 时，会发生：

```text
ReferenceError: content is not defined
```

错误处理本身失败，用户仍停留在“加载中”或看到空白页面。

#### 整改建议

1. `setState()` 内部通过 `document.getElementById('md-content')` 获取元素，或把元素作为参数传入。
2. 增加 401、403、404、500、空 Markdown、非法 JSON 和网络异常测试。
3. 保留 `textContent` 输出错误文本，避免重新引入 XSS。

#### 验收标准

- 所有失败分支均显示可读状态信息。
- 控制台无未捕获异常。
- 错误消息不能执行 HTML 或脚本。

## 五、仍未完全关闭的既有问题

### P1-5 密码哈希强度和同步阻塞

当前仍使用同步 PBKDF2 10,000 次。时序安全比较已落实，但同步低迭代哈希仍可能阻塞 Node 事件循环，抗离线破解能力也偏弱。

建议采用带版本字段的 Argon2id 或 scrypt 格式，并在用户成功登录后自动升级旧哈希。

### P1-6 代理、TLS 与内部错误泄露

- `trust proxy` 默认仍为 1；如果应用被直接暴露，攻击者可能伪造 `X-Forwarded-For`。
- PostgreSQL SSL 仍使用 `rejectUnauthorized:false`，未校验服务端证书。
- IPO、账户、导入和管理接口中仍有多处直接返回 `e.message`，绕过统一生产错误处理。
- README 建议公网开放 3000 端口，与 Nginx 部署说明冲突。

建议默认关闭 `trust proxy`，由生产配置显式启用；数据库配置 CA 校验；500 响应仅返回通用错误和请求 ID，内部异常写入服务端日志。

### P1-7 验证码联合限流不足

邮箱验证码仅按当前 Session 限制 60 秒。攻击者可更换 Cookie 或 Session，持续向同一邮箱发送邮件。

建议增加 IP、邮箱和设备维度的 Redis 联合限流，并设置日配额、失败审计和异常告警。

### P2-1 测试入口和产品规则不一致

`npm run test:all` 只收集 `server/test/*.test.js`，未纳入：

- `test-security.js`
- `ipo-report/test_ipo_frontend.js`
- Python 功能测试
- PostgreSQL 集成测试

IPO 前端独立测试当前 22/24，通过失败项为：

- `申购日未来->发行公告`
- `进展公告日=发行公告日(非未来)`

2026-07-16 的实现注释明确要求未来申购也展示“申购日”，因此应先确认产品规则，再同步代码或测试，不能长期保留失败测试。

### P2-2 CI 不能阻断部分回归和高危依赖

当前 CI：

- 只运行 `npm run test:security`，不运行 `npm run test:all`。
- IPO 前端测试未进入 CI。
- Python 只做语法编译，不安装依赖、不执行功能测试。
- `npm audit --audit-level=high || true` 永远不阻断构建。
- 未配置秘密扫描。
- GitHub Actions 仅固定到大版本标签，未固定完整提交 SHA。

本地分支比 `origin/master` 超前 8 个提交，当前新增迁移回归尚未经过远端 CI。现有集成测试在空 PostgreSQL 上调用 `initSchema()`，推送后很可能直接暴露 P0-1。

### P2-3 依赖与可复现环境

- Python `requirements.txt` 未锁定版本，也没有锁文件。
- `package.json` 未声明 Node `engines`。
- README 仍安装 Node 18，而 CI 和正式部署文档使用 Node 22。
- README 使用 `npm install --production`，建议与 lockfile 配合使用 `npm ci --omit=dev`。

### P2-4 仓库与部署文档残留

Docker 运行文件已经全部删除，但仍存在以下文字残留：

- `deploy/.env.example` 仍称数据库密码会供 `docker-compose` 使用。
- README 仍写“容器/PM2”。
- README 建议腾讯云安全组放行 3000，而正式部署文档要求只开放 80/443/22。
- `public/changelog.json` 包含历史 Docker 条目；历史记录可以保留，但应避免让读者误以为当前仍支持 Docker。

### P2-5 备份方案仍缺少生产闭环

`scripts/backup_db.sh` 已实现本地 `pg_dump + gzip` 和保留最近 14 份，但仍缺少：

- 备份加密。
- 异地存储。
- 完整性校验。
- 自动恢复演练。
- 明确的 RPO、RTO 和删除策略。

生产备份中包含全部用户和投资数据，不能只依赖普通目录下的明文压缩文件。
