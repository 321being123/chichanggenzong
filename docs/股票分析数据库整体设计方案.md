# 股票分析数据库整体设计方案

更新时间：2026-07-19  
状态：六个阶段全部完成；采集和读取均已切换新库，股票分析旧表已删除

## 一、设计目标

本方案用于解决当前股票分析功能不断增加字段后出现的三个问题：

1. 基础资料、行业、实际控制人等数据仍混在 `stock_analysis_stocks.data` JSON 中，无法可靠查询历史变化。
2. 日行情、估值、股本和市值混在 `stock_daily_valuations`，来源和单位不够清晰。
3. 大量最终指标只存在分析快照 JSON 中，无法按指标、日期和公式版本查询。

最终目标：

- 支持A股、港股，后续可扩展基金、债券、可转债、指数和美股。
- 支持Tushare、腾讯、巨潮、东方财富及后续数据源。
- 财报更正、行业调整、实际控制人变更等历史版本全部保留。
- 原始数据、标准化数据和计算结果分层存储。
- 页面不直接依赖任何一个数据源的字段名称。
- 新增指标时尽量只增加指标定义和计算逻辑，不反复修改核心表。
- 保持现有持仓、交易、打新日历和可转债业务不受影响。

## 二、参考项目与采用原则

本方案参考以下高关注金融项目的公开架构：

- [OpenBB](https://github.com/OpenBB-finance/OpenBB)：采用多数据提供商接入、统一标准模型输出。本项目借鉴“数据源适配层与业务模型分离”。
- [Microsoft Qlib](https://github.com/microsoft/qlib)：区分本地数据、共享数据服务和缓存，重视批量读取效率。本项目借鉴“原始事实、标准化事实、计算缓存分层”。
- [VeighNa/vn.py](https://github.com/vnpy/vnpy)：通过统一数据库接口适配PostgreSQL、SQLite和时序数据库。本项目借鉴“存储实现不进入业务计算逻辑”。
- [QuantConnect LEAN](https://github.com/QuantConnect/Lean)：使用稳定的证券身份管理不同市场和代码变化。本项目借鉴“内部证券ID与外部代码分离”。

不照搬这些项目的全部复杂度。当前继续使用单个PostgreSQL，不引入微服务、Kafka、ClickHouse或独立时序数据库。

## 三、总体分层

建议在PostgreSQL中使用六个逻辑区域：

| 区域 | 作用 | 主要内容 |
|---|---|---|
| `core` | 证券和公司主数据 | 证券、公司、代码、交易所、币种、行业、控制人 |
| `market` | 市场时序事实 | 日线、当前行情、估值、股本、市值、汇率 |
| `fundamental` | 公司基本面事实 | 财报、财务科目、分红、业绩预告、快报 |
| `event` | 公告和公司事项 | 巨潮公告、正式事项、论坛热点、来源关联 |
| `analytics` | 计算结果和展示缓存 | 指标定义、指标值、分位点、分析快照 |
| `ops` | 采集和质量控制 | 数据源、采集批次、同步游标、质量问题 |

现有用户、账户、持仓、交易等表继续保留在当前区域，首期不重构。

数据流：

```text
外部接口
  → 原始记录与采集批次
  → 标准化主数据/行情/财务事实
  → 指标计算结果
  → 页面分析快照
```

## 四、核心设计规则

### 1. 证券ID与股票代码分离

内部统一使用 `instrument_id BIGINT` 关联，不能继续用 `ts_code` 作为所有表的长期主键。

原因：

- 港股、A股、基金和债券的代码规则不同。
- 同一家公司可能同时有A股和H股。
- 股票可能更名、换代码、转板或退市。
- Tushare、腾讯、东方财富使用的代码格式不同。

`000001.SZ` 只是一个证券代码，不是永久公司身份。

### 2. 公司与证券分离

“平安银行股份有限公司”是公司；“000001.SZ”是该公司发行的一只证券。实际控制人、财报属于公司，价格、市值和交易状态属于证券。

### 3. 三种时间必须分开

- `period_end`：数据属于哪个报告期。
- `announced_at`：市场什么时候知道该数据。
- `ingested_at`：系统什么时候抓到该数据。

分析某个历史日期时，只允许使用当日之前已经公告的数据，避免使用未来信息。

### 4. 原始数据永不覆盖

同一份财报的原版、更正、重述都保留。标准化层标记当前有效版本，但不删除旧版本。

### 5. 单位统一

- 金额统一存最小货币单位口径的“元”，并保存 `currency_code`。
- 股本统一存“股”，不存“万股”。
- 比率统一存小数，`0.05` 表示 `5%`。
- 日期使用PostgreSQL `DATE`，不再使用 `TEXT YYYYMMDD`。
- 财务金额使用 `NUMERIC`，不使用浮点数。

### 6. JSON只做补充

经常筛选、排序、计算的字段必须是正式列。JSONB仅保存：

- 上游完整原始响应。
- 暂未标准化的低频字段。
- 指标诊断和展示说明。

## 五、建议新增表

### A. `ops` 数据源与采集层

#### `ops.data_sources`

记录数据源，不在业务表中散落字符串。

关键字段：

- `source_id`
- `source_code`：`tushare`、`tencent`、`cninfo`、`eastmoney`、`xueqiu`、`guba`
- `source_name`
- `source_type`：正式、行情、市场讨论
- `priority`
- `enabled`

#### `ops.ingestion_runs`

每次接口请求或批量同步的执行记录。

关键字段：

- `run_id`
- `source_id`
- `dataset_code`
- `started_at`、`finished_at`
- `request_range`
- `row_count`
- `status`
- `error_message`

#### `ops.raw_records`

保存上游原始记录，用于追溯和重新标准化。此表是落地层，不直接用于页面查询。

关键字段：

- `raw_record_id`
- `run_id`
- `source_id`
- `dataset_code`
- `source_key`
- `source_updated_at`
- `payload JSONB`
- `payload_hash`
- `ingested_at`

唯一键：`(source_id, dataset_code, source_key, payload_hash)`。

#### `ops.sync_cursors`

替代和扩展当前 `stock_data_sync_state`。

关键字段：

- `instrument_id`或`company_id`
- `dataset_code`
- `last_success_date`
- `last_source_update`
- `last_attempt_at`
- `last_error`
- `retry_count`

#### `ops.data_quality_issues`

记录缺失、异常和补算，不把问题只写在页面文字中。

例如：

- Tushare ROA为空。
- 流通股本缺失。
- 财报报告期不连续。
- 分红有每股金额但无法获得基准股本。
- 指标使用补算值。

### B. `core` 主数据层

#### `core.instruments`

所有可交易或可分析证券的统一主表。

关键字段：

- `instrument_id`
- `canonical_code`
- `name`
- `asset_class`：股票、基金、债券、可转债、指数
- `market`
- `exchange_code`
- `currency_code`
- `list_date`、`delist_date`
- `status`

唯一键：`canonical_code`。

#### `core.instrument_identifiers`

保存不同数据源代码及历史代码。

关键字段：

- `instrument_id`
- `source_id`
- `identifier_type`
- `identifier_value`
- `valid_from`、`valid_to`

例如同一证券可同时保存 `000001.SZ`、`sz000001`、`0.000001`。

#### `core.companies`

公司法人主体。

关键字段：

- `company_id`
- `legal_name`
- `short_name`
- `country_code`
- `registration_code`
- `company_type`
- `registered_capital`

#### `core.company_instruments`

连接公司和证券，支持一家公司发行A股、H股、债券和可转债。

唯一键：`(company_id, instrument_id, valid_from)`。

#### `core.industry_taxonomies`

行业标准版本，例如：

- 申万2021
- 申万旧版
- 国证行业
- 中证行业

#### `core.industry_nodes`

行业树节点。

关键字段：

- `taxonomy_id`
- `industry_code`
- `industry_name`
- `level`
- `parent_id`

#### `core.company_industry_memberships`

保存行业归属历史，而不是只保留当前行业。

关键字段：

- `company_id`
- `industry_node_id`
- `valid_from`、`valid_to`
- `announced_at`
- `source_id`
- `is_current`

#### `core.company_controllers`

保存实际控制人历史。

关键字段：

- `company_id`
- `controller_name`
- `controller_type`：自然人、中央国资、地方国资、集体、企业、无实际控制人、其他
- `control_ratio`
- `valid_from`、`valid_to`
- `announced_at`
- `source_id`
- `source_document_id`
- `confidence`

页面展示实际控制人时从此表选择当前有效记录，不再从股票基础JSON读取。

### C. `market` 行情与股本层

#### `market.daily_bars`

只保存OHLCV行情，不混入PE、PB和股本。

关键字段：

- `instrument_id`
- `trade_date`
- `open`、`high`、`low`、`close`
- `volume`
- `amount`
- `source_id`
- `ingested_at`

唯一键：`(instrument_id, trade_date, source_id)`。

#### `market.adjustment_factors`

保存复权因子及公司行为调整信息。

唯一键：`(instrument_id, trade_date, source_id)`。

#### `market.daily_valuations`

保存每日估值，不保存行情K线。

关键字段：

- `instrument_id`
- `trade_date`
- `pe_static`、`pe_ttm`
- `pb`
- `dividend_yield_ttm`
- `total_market_cap`
- `circulating_market_cap`
- `free_float_market_cap`
- `currency_code`
- `source_id`

#### `market.share_capital_history`

股本必须单独成表，因为它会随送转、增发、回购和解禁变化。

关键字段：

- `instrument_id`
- `effective_date`
- `total_shares`
- `a_shares`
- `h_shares`
- `circulating_shares`
- `free_float_shares`
- `restricted_shares`
- `source_id`

市值统一通过“价格×对应股本”计算；上游直接给出的市值同时保留在 `market.daily_valuations` 用于核验。

#### `market.latest_quotes`

替代仅面向腾讯的缓存表，保存当前行情的最后有效值。

关键字段：

- `instrument_id`
- `price`
- `currency_code`
- `quote_time`
- `source_id`
- `is_stale`
- `fetched_at`

#### `market.fx_rates`

为港股和多币种账户预留。

唯一键：`(base_currency, quote_currency, rate_date, source_id)`。

### D. `fundamental` 财务与公司行为层

#### `fundamental.financial_reports`

财报版本头表，统一管理三表。

关键字段：

- `report_id`
- `company_id`
- `period_end`
- `period_type`：一季报、半年报、三季报、年报
- `statement_scope`：合并、母公司
- `announced_at`
- `source_id`
- `source_version`
- `update_flag`
- `is_current_version`
- `raw_record_id`

同报告期更正版本新增一行，不覆盖旧版本。

#### `fundamental.financial_facts`

统一保存利润表、资产负债表、现金流量表和财务指标的标准化科目。

关键字段：

- `report_id`
- `statement_type`
- `metric_code`
- `numeric_value`
- `text_value`
- `unit_code`
- `currency_code`
- `source_field`
- `is_calculated`

唯一键：`(report_id, metric_code)`。

常用 `metric_code` 示例：

- `net_profit_parent`
- `net_profit_deducted`
- `total_assets`
- `equity_parent`
- `operating_cashflow`
- `goodwill`
- `interest_expense`
- `roe`
- `roa`

这样新增国际财报字段时不必不断给三张宽表增加列。高频查询通过物化视图解决，不直接在页面逐行聚合事实表。

#### `fundamental.financial_period_summary`

常用财务字段的查询加速表，由 `financial_facts` 生成。

保存页面最常用的归母利润、扣非利润、净资产、总资产、经营现金流、商誉、ROE、ROA等。它是可重建缓存，不是原始事实。

#### `fundamental.corporate_actions`

统一保存现金分红、送股、转增、拆并股、配股、增发和回购。

关键字段：

- `action_id`
- `instrument_id`
- `action_type`
- `fiscal_period_end`
- `announced_at`
- `record_date`
- `ex_date`
- `pay_date`
- `status`
- `cash_per_share_pre_tax`
- `cash_per_share_after_tax`
- `stock_dividend_ratio`
- `capitalization_ratio`
- `base_shares`
- `total_cash_amount`
- `currency_code`
- `source_id`

没有分红的年度不伪造为公司行为记录；页面中的“未分红”由年度财报和公司行为表关联后生成。

#### `fundamental.earnings_guidance`

保存业绩预告、业绩快报和管理层指引。

关键字段：

- `company_id`
- `period_end`
- `guidance_type`
- `announced_at`
- `profit_min`、`profit_max`
- `change_min`、`change_max`
- `summary`
- `change_reason`
- `currency_code`
- `source_id`

### F. `analytics` 指标与快照层

#### `analytics.metric_definitions`

所有计算指标的说明书。

关键字段：

- `metric_code`
- `metric_name`
- `category`
- `value_type`
- `unit_code`
- `formula_text`
- `formula_version`
- `negative_rule`
- `zero_rule`
- `enabled`

例如：滚动PE、累计分红率、十年PE分位、上市至今年化收益率、净现金安全额。

#### `analytics.metric_values`

按证券、计算日期和公式版本保存指标结果。

关键字段：

- `instrument_id`
- `metric_code`
- `as_of_date`
- `period_start`、`period_end`
- `numeric_value`
- `text_value`
- `status`：有效、无法计算、缺少数据、补算
- `formula_version`
- `input_hash`
- `diagnostics JSONB`
- `calculated_at`

唯一键：`(instrument_id, metric_code, as_of_date, period_start, period_end, formula_version)`。

时间筛选后的累计分红率可以保存为区间指标，也可以即时计算；常用3年、5年、10年区间可以缓存。

#### `analytics.metric_statistics`

保存分位点及有效样本说明。

关键字段：

- `instrument_id`
- `metric_code`
- `as_of_date`
- `window_start`、`window_end`
- `percentile_value`
- `valid_samples`
- `excluded_samples`
- `excluded_reason`

#### `analytics.analysis_snapshots`

保留最终页面快照，但快照不再是唯一数据来源。

关键字段：

- `instrument_id`
- `as_of_date`
- `snapshot_type`
- `formula_bundle_version`
- `payload JSONB`
- `source_watermark`
- `created_at`

#### `analytics.stock_overview_latest`

面向页面的最新概览缓存，可使用普通表或物化视图。

包含：

- 名称、代码、行业、上市日期、币种。
- 当前价格和四类市值。
- 实际控制人。
- 最新报告日期和业绩预告。
- 常用PE、PB、ROE、ROA和收益率。

页面加载概览时只查这一处，详情模块再按需读取其他表。

## 七、索引与容量策略

### 必要索引

- 行情：`(instrument_id, trade_date DESC)`。
- 财报：`(company_id, period_end DESC, announced_at DESC)`。
- 财务事实：`(report_id, metric_code)`。
- 公司行为：`(instrument_id, ex_date DESC)`和`(instrument_id, fiscal_period_end DESC)`。
- 公告：`(company_id, announced_at DESC)`。
- 指标：`(instrument_id, metric_code, as_of_date DESC)`。
- 同步状态：`(instrument_id, dataset_code)`。

### 分区原则

当前 `stock_daily_valuations` 约1.5万行，暂时不需要分区。满足以下任一条件后再对行情表按年份分区：

- 单表超过500万行。
- 增加全A股日线或分钟线。
- 常规查询和维护明显变慢。

不要现在提前引入TimescaleDB，避免增加部署和备份复杂度。

### 保留策略

- 财报、公司行为、公告、行业和控制人历史：永久保留。
- 日线行情和估值：永久保留。
- 实时行情缓存：只保留最新值；需要历史分钟线时另建表。
- 原始接口响应：正式财务和公司行为永久保留，行情原始响应可按期限清理。
- 分析快照：按日保留，重复手动刷新只保留最新一次或按版本去重。

## 八、数据更新与版本规则

### 首次建档

1. 建立证券和公司身份。
2. 抓取上市以来代码、行业、控制人和股本历史。
3. 抓取上市以来三表、分红和业绩预告。
4. 抓取所需行情、估值和复权因子。
5. 标准化并运行质量检查。
6. 计算指标并生成页面快照。

### 每日更新

- 当前行情：腾讯缓存约60秒。
- 日行情、估值、股本：交易日收盘后更新。
- 财报和公司行为：每天检查新增及更正。
- 行业、控制人：每日或每周检查，发生变化时关闭旧记录并新增记录。
- 公司大事：正式公告优先；市场讨论失败不影响正式分析。

### 失败处理

- 空结果不能覆盖上一份有效数据。
- 每次失败写入 `ops.ingestion_runs` 和 `ops.sync_cursors`。
- 部分字段缺失时保留完整行，写入质量问题。
- 补算指标必须设置 `status='calculated'`，并记录公式和输入。

## 九、分阶段实施计划

### 第一阶段：基础骨架

新增 `ops`、`core`、`market`、`fundamental`、`event`、`analytics` 六个schema以及：

- 数据源、采集批次、同步游标。
- 证券、代码、公司关系。
- 行业标准、行业归属、实际控制人。
- 指标定义表。

此阶段不切换现有页面。

### 第二阶段：行情和股本拆分

- 把日线、复权、估值、股本、市值拆分。
- 统一币种和单位。
- 回填现有 `stock_daily_valuations`。
- 新旧表双写并核对一段时间。

### 第三阶段：财报事实模型

- 建立财报头表和财务事实表。
- 回填三表及财务指标。
- 验证报告版本、更正版本、公告日期和单位。
- 建立财务期间汇总缓存。

### 第四阶段：公司行为和事件

- 迁移分红、送转、业绩预告、公告和公司大事。
- 补齐未分红年度的页面生成逻辑。
- 建立行业和控制人历史更新任务。

### 第五阶段：指标中心

- 把PE、PB、ROE、ROA、增长率、现金流、分位点等写入指标表。
- 对所有补算值增加状态、公式版本和输入来源。
- 建立页面概览缓存和分析快照。

### 第六阶段：切换与清理

- 接口切换到新表读取。
- 新旧结果逐股票核对。
- 保留兼容视图，停止旧表写入。
- 至少稳定运行一个完整财报周期后，再单独申请是否归档旧表。

任何阶段都不直接删除、重命名或覆盖现有核心表。

## 十、验收标准

- 同一公司A股、H股能共享财报、行业和控制人，同时拥有独立价格和市值。
- 代码变化后历史数据仍关联同一个证券身份。
- 财报更正版本不会覆盖历史版本。
- 任意指标都能追溯公式版本、数据来源和计算时间。
- 官方ROA为空但使用补算时，数据库能够明确区分官方值和补算值。
- 行业能明确显示申万或国证及级别；取不到级别时只显示行业名称。
- 总市值、A股市值、流通市值和自由流通市值能追溯到对应价格和股本。
- 港股价格和市值能保存港币，并可通过汇率折算。
- 未分红年度由年度财报与公司行为关联生成，不伪造上游分红记录。
- 外部接口空值或失败不会覆盖上一份有效数据。
- 现有持仓、交易、打新日历、可转债和用户自选不受影响。

## 十一、最终建议

下一步不要继续给 `stock_analysis_stocks.data` 或 `stock_analysis_snapshots.data` 增加核心字段。

建议先执行第一阶段，建立稳定的证券、公司、行业、控制人、数据来源和指标定义骨架；随后再迁移行情和财报。这样后续增加港股、更多市值口径、股东、管理层、机构持仓、回购、质押、诉讼和分析指标时，都有固定归属，不需要再次重做整个数据库。

## 十二、后续功能强制设计原则

1. 外部数据首次全量拉取后必须落库，页面优先读取数据库；后续使用同步游标和短重叠窗口增量更新，减少API调用。
2. 数据库必须为新增字段、数据源、市场、币种和历史版本预留扩展能力；正式列与原始 `JSONB` 合理冗余，保证查询效率和数据追溯。
3. 新增数据库或功能代码必须从整体数据及产品架构评估采集、存储、计算、接口、页面、调度、权限、缓存和现有功能兼容性，不得建立孤立或重复链路。

详细执行约束统一记录在项目根目录 `AGENTS.md`，后续开发必须遵守。
